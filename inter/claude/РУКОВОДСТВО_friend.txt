ОБЩАЯ ИДЕЯ
----------
Наша задача — написать программу на C++, которая:
  1. Читает текст программы на модельном языке из файла
  2. Разбирает его (понимает структуру)
  3. Выполняет его

Мы делаем это в 3 фазы:
  Фаза 1 — ЛЕКСЕР:    текст → список лексем (токенов)
  Фаза 2 — ПАРСЕР:    список токенов → ПОЛИЗ (внутренний код)
  Фаза 3 — ИНТЕРПР.:  ПОЛИЗ → выполнение программы

Все три фазы запускаются одна за другой в функции main().


================================================================
ШАГ 1. ЧТЕНИЕ ФАЙЛА
================================================================

Что нужно сделать:
  Прочитать весь текст программы из файла в строку.

Как делаем:
  - Принимаем имя файла как аргумент командной строки (argv[1])
  - Открываем через ifstream
  - Читаем всё содержимое в std::string через rdbuf()

Пример:
  ifstream f(argv[1]);
  ostringstream ss;
  ss << f.rdbuf();
  string src = ss.str();

Что получаем:
  Строка src содержит весь исходный текст.
  Например: "program { int a = 5; write(a); }"


================================================================
ШАГ 2. ЛЕКСИЧЕСКИЙ АНАЛИЗ (ЛЕКСЕР)
================================================================

Что такое лексема (токен)?
  Это минимальная смысловая единица языка.
  "for" — одна лексема (ключевое слово).
  "3.14" — одна лексема (вещественный литерал).
  "<=" — одна лексема (оператор).

Зачем нужен лексер?
  Чтобы дальнейший код работал не с символами, а со смысловыми
  единицами. Парсеру не важно сколько пробелов между словами —
  лексер их уже съел.

Алгоритм (конечный автомат):
  - Пропустить пробелы, переносы строк, табуляции
  - Встретили "/*" — это комментарий, пропускаем до "*/"
  - Встретили цифру [0-9] — читаем число
      Если потом идёт "." и ещё цифра — это ВЕЩЕСТВЕННОЕ (real)
      Иначе — это ЦЕЛОЕ (int)
  - Встретили букву или "_" — читаем идентификатор/ключевое слово
      Потом сравниваем с таблицей: for, step, until, do, if, else,
      while, goto, read, write, and, or, not, int, string, real
  - Встретили '"' — читаем строку до следующего '"'
  - Встретили "<=", ">=", "==", "!=" — двухсимвольный оператор
  - Встретили "+", "-", "*", "(", ";" и т.д. — односимвольный оператор
  - Встретили ":" — двоеточие (разделитель метки)

Новые токены по сравнению с базовым вариантом:
  KW_FOR   — for
  KW_STEP  — step
  KW_UNTIL — until
  KW_DO    — do
  KW_GOTO  — goto
  COLON    — : (двоеточие — разделитель метки)

Что получаем:
  Вектор токенов. Например для "for i = 1 step 1 until 5 do"
  получим: [KW_FOR] [ID "i"] [ASSIGN] [INT_LIT "1"] [KW_STEP]
           [INT_LIT "1"] [KW_UNTIL] [INT_LIT "5"] [KW_DO]


================================================================
ШАГ 3. ТАБЛИЦА СИМВОЛОВ
================================================================

Что хранит:
  - Переменные: имя → { тип, значение, инициализирована }
  - Метки (labels): имя метки → адрес в ПОЛИЗ

Зачем два словаря?
  Метки — это не переменные. У них нет типа и значения.
  Метка — это просто "закладка" в ПОЛИЗ.
  Контекстное условие: метка "объявляется" её использованием
  в помеченном операторе. Повторная метка — ошибка.

Что такое std::variant?
  Тип C++17 который хранит одно из нескольких значений.
  using Val = variant<long long, double, string>;
  В одном объекте либо целое число, либо вещественное, либо строка.

Правила совместимости типов (Таблица №2, без boolean):
  int op int   → int
  real op real → real
  int op real  → real (int расширяется до real)
  real op int  → real
  string + string → string (конкатенация)
  int = real   → усечение (3.9 → 3)
  real = int   → расширение (3 → 3.0)


================================================================
ШАГ 4. ПОЛИЗ — ВНУТРЕННИЙ КОД ПРОГРАММЫ
================================================================

Что такое ПОЛИЗ?
  Постфиксная запись — последовательность инструкций для стековой машины.

Пример: a + b * 2
  Обычная запись: a + b * 2
  ПОЛИЗ:          LOAD a  LOAD b  PUSH 2  MUL  ADD

  Выполнение на стеке:
    LOAD a    → [a]
    LOAD b    → [a, b]
    PUSH 2    → [a, b, 2]
    MUL       → [a, b*2]
    ADD       → [a+b*2]

Как реализован for I = E1 step E2 until E3 do S:

  По ТЗ семантика такая:
    - Вычислить E1, E2, E3 ОДИН РАЗ перед циклом
    - I = E1
    - Если I > E3 — не выполнять ни разу
    - Выполнить S, потом I = I + E2
    - Повторять пока I <= E3

  В ПОЛИЗ это выглядит так:
    [E1] → STORE I, POP        ← I = E1 (один раз)
    [E2] → STORE __step, POP   ← сохраняем шаг во временную переменную
    [E3] → STORE __limit, POP  ← сохраняем предел

  метка_check:
    LOAD I
    LOAD __limit
    GT                         ← I > limit?
    JNZ → метка_end            ← если да, выходим

    [S]                        ← тело цикла

    LOAD I
    LOAD __step
    ADD
    STORE I, POP               ← I = I + step

    JMP → метка_check
  метка_end:

  Зачем __step и __limit? Потому что E2 и E3 вычисляются
  ОДИН РАЗ до входа в цикл (требование ТЗ).
  Мы храним их в скрытых служебных переменных.

Как реализован goto:

  goto — это безусловный переход на метку.
  Проблема: goto может идти как ВПЕРЁД так и НАЗАД по коду.

  Случай 1 — goto на метку которая уже была объявлена (назад):
    Адрес метки уже известен → emit JMP с готовым адресом.

  Случай 2 — goto на метку которая ещё не встречена (вперёд):
    Адрес не известен → emit JMP с адресом 0, запомнить индекс.
    Когда метка встретится — "запатчить" (заполнить) адрес.

  Структура данных для меток:
    map<string, int>       labels;     // имя → адрес в ПОЛИЗ
    map<string, vector<int>> fwdGotos; // имя → список индексов JMP для патчинга

  Когда встречаем "метка : оператор":
    1. Проверяем что метка не объявлена дважды
    2. Запоминаем labels[name] = here()
    3. Патчим все "незакрытые" fwdGotos[name]

  Когда встречаем "goto метка":
    1. Если метка уже в labels → emit JMP с готовым адресом
    2. Иначе → emit JMP с адресом 0, добавляем индекс в fwdGotos[name]

  После полного разбора программы:
    Проверяем что fwdGotos пуст (все goto нашли свою метку).

Как реализованы ОБЫЧНЫЕ (не ленивые) вычисления and/or:

  В отличие от ленивых, здесь ВСЕГДА вычисляются оба операнда.
  Просто генерируем оба подвыражения, потом операцию:

    [левый]
    [правый]
    AND (или OR)

  Инструкции AND и OR в интерпретаторе:
    AND: снять R и L, положить (toBool(L) && toBool(R)) ? 1 : 0
    OR:  снять R и L, положить (toBool(L) || toBool(R)) ? 1 : 0


================================================================
ШАГ 5. ПАРСЕР — СИНТАКСИЧЕСКИЙ АНАЛИЗ + ГЕНЕРАЦИЯ ПОЛИЗ
================================================================

Метод рекурсивного спуска:
  Каждому правилу грамматики соответствует один метод.
  Вызовы образуют дерево разбора.

Приоритет операций (цепочка методов):
  parseExpr → parseAssign → parseOr → parseAnd →
  parseRel → parseAdd → parseMul → parseUnary → parsePrimary

Как разобрать помеченный оператор и goto:

  В parseStmt() перед обычными операторами проверяем:
  "видим ID и следующий токен ':'?"
  Если да — это метка.

  Псевдокод:
    if (check(ID) && peekNext() == COLON):
        name = eat(ID)
        eat(COLON)
        // регистрируем метку
        if (labels.count(name)) → ошибка "метка объявлена дважды"
        labels[name] = here()
        // патчим все forward-goto на эту метку
        for idx in fwdGotos[name]: patch(idx, here())
        fwdGotos.erase(name)
        // разбираем помеченный оператор
        parseStmt()

  Для goto:
    if tryEat(KW_GOTO):
        name = eat(ID).val
        eat(SEMICOLON)
        if (labels.count(name)):
            emit JMP с адресом labels[name]    ← уже знаем адрес
        else:
            idx = emit JMP с адресом 0
            fwdGotos[name].push_back(idx)      ← запомним для патчинга

Как разобрать for I = E1 step E2 until E3 do S:

  1. Съесть KW_FOR
  2. name = eat(ID) — имя параметра цикла
  3. eat(ASSIGN)
  4. Генерируем E1: parseExpr()
     emit STORE name, emit POP         ← I = E1
  5. eat(KW_STEP)
  6. Генерируем E2: parseExpr()
     stepVar = "__step_" + уникальный суффикс
     sym.declare(stepVar, INT)
     emit STORE stepVar, emit POP      ← сохраняем шаг
  7. eat(KW_UNTIL)
  8. Генерируем E3: parseExpr()
     limitVar = "__limit_" + суффикс
     sym.declare(limitVar, INT)
     emit STORE limitVar, emit POP     ← сохраняем предел
  9. eat(KW_DO)
  10. loopStart = here()
  11. Проверка: LOAD name; LOAD limitVar; GT; JNZ → конец
      (если I > limit — выходим)
  12. parseStmt() — тело цикла
  13. LOAD name; LOAD stepVar; ADD; STORE name; POP
      ← I = I + step
  14. emit JMP loopStart
  15. patch адрес выхода


================================================================
ШАГ 6. ИНТЕРПРЕТАТОР (СТЕКОВАЯ ВИРТУАЛЬНАЯ МАШИНА)
================================================================

Выполняет ПОЛИЗ последовательно.
pc (program counter) — индекс текущей инструкции.

Новые опкоды по сравнению с базовым:
  AND — снять R, L; положить (toBool(L) && toBool(R)) ? 1 : 0
  OR  — снять R, L; положить (toBool(L) || toBool(R)) ? 1 : 0
  JNZ — снять со стека; если НЕ 0 → прыгнуть (для for: I > limit)

  (JMP и JZ остаются как были)

Арифметика с типами (Таблица №2):
  int op int   → int
  real op real → real  (или real op int, int op real → real)
  string + string → конкатенация


================================================================
ШАГ 7. СБОРКА В MAIN
================================================================

int main(int argc, char* argv[]) {
    // 1. Читаем файл
    string src = readFile(argv[1]);

    // 2. Лексический анализ
    Lexer lexer(src);
    vector<Token> tokens = lexer.tokenize();

    // 3. Парсинг + генерация ПОЛИЗ
    SymTable sym;
    vector<PolizOp> code;
    Parser parser(tokens, sym, code);
    parser.parse();

    // 4. Интерпретация
    Interpreter interp(code, sym);
    interp.run();
}

Все ошибки ловятся через try/catch(runtime_error).


================================================================
СТРУКТУРА ПРОГРАММЫ (всё в одном файле main.cpp)
================================================================

  Часть 1 — TT (enum class) и Token (struct)
  Часть 2 — Lexer (класс)
  Часть 3 — VType, Val, SymTable (классы)
  Часть 4 — OpCode (enum class) и PolizOp (struct)
  Часть 5 — Parser (класс с рекурсивным спуском)
  Часть 6 — Interpreter (класс, стековая ВМ)
  Часть 7 — main()

Сборка:
  g++ -std=c++17 -Wall -o interp main.cpp

Запуск:
  ./interp program.ml
  ./interp program.ml < input.txt


================================================================
КЛЮЧЕВЫЕ ОТЛИЧИЯ ОТ ВАРИАНТА С ЛЕНИВЫМИ ВЫЧИСЛЕНИЯМИ
================================================================

1. AND и OR — ВСЕГДА вычисляют оба операнда.
   Вместо цепочки JZ/JMP просто генерируем оба подвыражения,
   потом инструкцию AND или OR.

2. for...step...until...do — вместо do-while/break.
   Сложнее: нужны скрытые переменные для шага и предела,
   потому что E2 и E3 вычисляются ОДИН РАЗ до входа в цикл.

3. goto с метками — вместо break.
   Двунаправленные прыжки: нужен механизм "forward goto"
   (патчинг незакрытых переходов).
   Метки объявляются их использованием (имя: оператор).
   Повторная метка — ошибка (контекстное условие).

4. while остаётся (он в общей части ТЗ).


================================================================
ГЛОССАРИЙ ТЕРМИНОВ
================================================================

Лексема (токен)  — минимальная смысловая единица: слово, число, символ
Лексер           — разбивает текст на лексемы
КА               — конечный автомат, модель лексера
Парсер           — проверяет грамматику и строит внутренний код
Рекурсивный      — каждое правило грамматики = метод класса
спуск
ПОЛИЗ            — Польская Инверсная Запись, постфиксная форма
Стековая машина  — выполняет ПОЛИЗ используя стек
Таблица символов — хранит переменные и метки
Патчинг          — заполнение адресов переходов постфактум
Forward goto     — goto на метку которая ещё не встречена
std::variant     — тип C++17: хранит одно из нескольких значений
std::runtime_err — исключение C++: сообщение об ошибке


================================================================
ТИПИЧНЫЕ ОШИБКИ И КАК ИХ ДИАГНОСТИРОВАТЬ
================================================================

Тип ошибки              Когда возникает              Где ловить
----------------------  ---------------------------  ----------
Лексическая             Неизвестный символ           Lexer
Синтаксическая          Пропущена ; или )            Parser (eat)
Семантическая           Переменная не объявлена      SymTable
Семантическая           Повторное объявление         SymTable
Семантическая           Метка объявлена дважды       Parser
Семантическая           goto на необъявленную метку  Parser (конец parse)
Семантическая           Несовместимость типов        SymTable (set)
Времени выполнения      Деление на ноль              Interpreter
Времени выполнения      Не инициализирована          SymTable (value)

Все ошибки бросают throw std::runtime_error("сообщение")
и ловятся в main() одним блоком catch.
